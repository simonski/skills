# QA

**Score: 45/100** (was 55)

## What is being assessed
QA covers test breadth, isolation, risk concentration, and whether the highest-value user paths are protected by automated checks. Good looks like strong coverage of command behavior, deterministic tests, and CI gates that reflect the user-facing surface.

## Methodology
Ran the existing Go test and coverage workflow locally, counted test files/functions, and read the current test suites. Files reviewed included `cmd/update_test.go`, `internal/catalog/catalog_test.go`, `internal/project/project_test.go`, and `internal/version/version_test.go`.

## Findings

### Passing checks
- The repo has 4 test files and 27 test functions covering catalog, project, update flows, and version helpers (`cmd/update_test.go:20-163`, `internal/catalog/catalog_test.go:9-130`, `internal/project/project_test.go:11-98`, `internal/version/version_test.go:9-41`).
- Filesystem tests use `t.TempDir()` for isolation (`cmd/update_test.go:20-29`, `internal/project/project_test.go:11-17,34-36,45-47`).
- Tests avoid sleep-based timing and external service setup; the suites are small and deterministic (`cmd/update_test.go:20-163`, `internal/project/project_test.go:11-98`).

### Issues found
| Finding | Severity | Location | Recommendation |
|---|---|---|---|
| Most user-facing command runners are untested | High | `cmd/add.go:36-75`, `cmd/get.go:24-30`, `cmd/ls.go:29-66`, `cmd/rm.go:25-37`, `cmd/search.go:24-55`, `cmd/version.go:21-40`, `cmd/versions.go:25-38` | Add unit tests for each `run*` function, especially error cases and stdout/stderr behavior. |
| `runInit` is the largest command path and has no tests | High | `cmd/init.go:93-308` | Split selection logic from terminal I/O and test the extracted pieces directly. |
| Version HTTP behavior is barely covered; the tests never exercise `LatestRelease` | Medium | `internal/version/version.go:19-45`, `internal/version/version_test.go:9-41` | Inject an HTTP client or use `httptest.Server` so update checks can be tested without live network calls. |
| Current total coverage is 37.8%, and `cmd` coverage is only 13.9% | Medium | `.github/workflows/publish.yml:20-24` | Prioritize command-level coverage before raising the CI threshold further. |

## Verdict
The internal packages have decent coverage, but the command layer is still the weak point. The most important user paths remain under-tested, which is why QA drops despite better CI plumbing.

## Changes since last assessment
- The repo now has a coverage gate in CI (`.github/workflows/publish.yml:20-24`).
- Coverage quality did not keep pace with feature growth; `cmd` remains thinly tested.
